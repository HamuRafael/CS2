# AI Agent Project Specification: CS2 Stats Comparison Website

## 1. Project Overview

Build a web application that allows users to compare Counter-Strike 2 (CS2) player statistics using Steam profile URLs and the existing Leetify Public API.

The user will enter two Steam profile URLs:

1. Their own Steam URL
2. The Steam URL of the player they want to compare against

The system will fetch available CS2 statistics from the Leetify API, normalize the data, store useful comparison history in MySQL, and present a clear visual comparison in the frontend.

## 2. Tech Stack

### Backend

- Flask

### Frontend

- Vue.js

- Tailwind CSS or another modern CSS framework

### External API

- Documentation: `https://api-public-docs.cs-prod.leetify.com/`

## 3. Core Goal

Create a CS2 statistics comparison platform where a user can paste two Steam URLs and receive an easy-to-understand comparison of both players based on available Leetify data.

The comparison should focus on performance indicators such as aim, utility, clutching, opening duels, win rate, rating, map performance, and recent match trends, depending on the data exposed by the Leetify API.

## 4. User Flow

1. User opens the website.
2. User sees a form with two input fields:
   - `Your Steam Profile URL`
   - `Opponent / Friend Steam Profile URL`
3. User clicks `Compare`.
4. Frontend sends both URLs to the Flask backend.
5. Backend validates and parses the Steam URLs.
6. Backend resolves or extracts the required Steam identifier.
7. Backend requests player data from the Leetify Public API.
8. Backend normalizes the API response into a comparison-friendly format.
9. Backend stores the comparison request and relevant metadata in MySQL.
10. Frontend displays:
    - Player summary cards
    - Side-by-side stat comparison
    - Charts
    - Difference indicators
    - Match/recent performance insights

## 5. Functional Requirements

### 5.1 Steam URL Input

The site must accept Steam profile URLs in common formats, including:

- `https://steamcommunity.com/profiles/{steamId64}`
- `https://steamcommunity.com/id/{customName}`

The backend must validate that both URLs are present and appear to be valid Steam profile URLs.

If a custom Steam URL is provided, the backend should resolve it to a SteamID64 if required by the Leetify API. If the Leetify API can accept a Steam URL or custom ID directly, use the API-supported method.

### 5.2 Player Data Retrieval

The backend must call the Leetify Public API to retrieve available CS2 statistics for each player.

The backend must handle:

- Player found
- Player not found
- Player profile private or unavailable
- User not registered on Leetify
- API timeout
- API rate limit
- Invalid Steam URL
- Unexpected API response format

### 5.3 Stats Comparison

The application should compare available metrics such as:

- Overall rating
- Win rate
- K/D ratio
- Headshot percentage
- Aim rating
- Utility rating
- Clutch rating
- Opening duel performance
- Trade performance
- Map-specific performance
- Recent match performance
- Number of matches analyzed

The exact fields must be adapted to the response structure provided by the Leetify API.

### 5.4 Comparison Output

The frontend must display:

- Player A card
- Player B card
- Main stat comparison table
- Visual charts for key metrics
- Highlighted winner per metric
- Summary text generated by business logic, for example:
  - “Player A has stronger aim performance.”
  - “Player B has better utility impact.”
  - “Both players have similar win rates.”

### 5.5 History

The backend should store previous comparisons in MySQL.

Stored data should include:

- Comparison ID
- Player A Steam ID
- Player B Steam ID
- Player A display name if available
- Player B display name if available
- Timestamp
- Normalized comparison result as JSON

The app may expose a history page where users can view recent comparisons.

## 6. Non-Functional Requirements

### Performance

- API responses should be cached when possible to avoid repeated calls for the same player.
- The backend should avoid calling the external API unnecessarily.
- Comparison response time should be optimized for a smooth user experience.

### Security

- Validate all user inputs.
- Do not expose backend API keys or secrets to the frontend.
- Store API keys in environment variables.
- Use CORS rules instead of allowing all origins in production.
- Sanitize URLs and user-provided values.
- Use prepared queries or ORM-based database access.

### Reliability

- Add proper error handling for external API failures.
- Log backend errors without exposing sensitive details to users.
- Return clear API error messages to the frontend.

### Scalability

- Separate API routes, services, models, and utility functions.
- Keep Leetify API integration isolated in its own service module.
- Make the project easy to extend with new comparison metrics.

## 9. REST API Design

### POST `/api/compare`

Compares two players.

#### Request Body

```json
{
  "playerAUrl": "https://steamcommunity.com/profiles/12345678901234567",
  "playerBUrl": "https://steamcommunity.com/id/example"
}
```

#### Successful Response

```json
{
  "comparisonId": 1,
  "players": {
    "playerA": {
      "steamId": "12345678901234567",
      "name": "Player A",
      "avatar": "https://example.com/avatar-a.png"
    },
    "playerB": {
      "steamId": "76561198000000000",
      "name": "Player B",
      "avatar": "https://example.com/avatar-b.png"
    }
  },
  "stats": [
    {
      "metric": "Headshot Percentage",
      "playerA": 48.2,
      "playerB": 42.7,
      "winner": "playerA"
    }
  ],
  "summary": [
    "Player A has better aim-related metrics.",
    "Player B performs better in utility impact."
  ]
}
```

### GET `/api/comparisons`

Returns recent comparison history.

### GET `/api/comparisons/{comparisonId}`

Returns a specific comparison.

### GET `/api/players/{steamId}`

Returns cached player data if available.

## 10. Database Schema

### `players`

| Column | Type | Description |
|---|---|---|
| id | INT PK | Internal ID |
| steam_id | VARCHAR(32) UNIQUE | SteamID64 |
| steam_url | VARCHAR(255) | Original Steam URL |
| display_name | VARCHAR(120) | Player name if available |
| avatar_url | VARCHAR(255) | Player avatar if available |
| last_fetched_at | DATETIME | Last API fetch timestamp |
| raw_data | JSON | Raw or partially normalized API response |
| created_at | DATETIME | Creation timestamp |
| updated_at | DATETIME | Update timestamp |

### `comparisons`

| Column | Type | Description |
|---|---|---|
| id | INT PK | Internal comparison ID |
| player_a_id | INT FK | Reference to player A |
| player_b_id | INT FK | Reference to player B |
| result_json | JSON | Normalized comparison result |
| created_at | DATETIME | Comparison timestamp |

## 11. Leetify API Integration Rules

The backend must be the only layer that communicates with Leetify.

The frontend must never call the Leetify API directly.

Create a dedicated service:

```python
class LeetifyService:
    def get_player_profile(self, steam_id: str) -> dict:
        pass

    def get_player_stats(self, steam_id: str) -> dict:
        pass

    def get_recent_matches(self, steam_id: str) -> dict:
        pass
```

The actual endpoint names and response fields must be implemented according to the official Leetify Public API documentation.

## 12. Data Normalization

Create a normalized internal format so the frontend does not depend directly on the external API response.

Example normalized stat object:

```json
{
  "key": "headshot_percentage",
  "label": "Headshot Percentage",
  "category": "Aim",
  "playerAValue": 48.2,
  "playerBValue": 42.7,
  "unit": "%",
  "higherIsBetter": true,
  "winner": "playerA"
}
```

## 13. Comparison Logic

For each metric:

1. Check if both players have the metric.
2. Normalize value types.
3. Determine whether higher or lower is better.
4. Calculate difference.
5. Select winner.
6. Generate short insight.

Example:

```text
If Player A headshot percentage is 48.2 and Player B is 42.7, Player A wins that metric by 5.5 percentage points.
```

## 20. AI Agent Instructions

When generating code for this project, the AI agent must:

1. Keep backend and frontend separated.
2. Use Flask for the backend REST API.
3. Use Vue.js for the frontend.
4. Use MySQL for persistence.
5. Never expose API keys in frontend code.
6. Keep Leetify API calls inside backend service modules.
7. Normalize external API responses before sending data to the frontend.
8. Include clear error handling.
9. Use reusable Vue components.
10. Avoid hardcoding API response fields until verified against official Leetify documentation.
11. Add comments only where they improve maintainability.
12. Prefer clean, simple, beginner-friendly architecture.
